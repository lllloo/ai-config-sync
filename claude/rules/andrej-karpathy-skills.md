# 範圍與驗證規則

減少「做超過要求」與「未驗證就斷言」兩類常見錯誤。改編自 andrej-karpathy-skills 的 CLAUDE.md，與本機既有規則合併適用。

**取捨**：這些規則偏向謹慎而非速度。瑣碎任務用判斷力，不必逐條套。

## 動手前先想

不臆測、不藏困惑、把取捨攤開：

- 假設要明講；不確定就問，不要靜默選一種解讀。
- 有多種解讀時全部列出，不代使用者決定。
- 有更簡單的做法就說；該 push back 時就 push back。
- 有東西不清楚就停下，指出卡在哪，然後問。

## 最小改動

只寫解決問題的最少程式碼，不做投機性擴充：

- 不加沒被要求的功能、抽象、彈性、可設定性。
- 不為單次使用的程式碼做抽象。
- 不為不可能發生的情境寫錯誤處理。
- 寫了 200 行但 50 行能解決，就重寫。
- 自問：資深工程師會不會覺得這過度複雜？會，就簡化。

## 外科手術式修改

只碰非碰不可的，只清自己製造的垃圾：

- 不「順手改善」相鄰的程式碼、註解、格式。
- 不重構沒壞的東西；沿用既有風格，即使你會做法不同。
- 看到無關的 dead code 就**指出**，不擅自刪。
- 自己的改動造成的孤兒 import／變數才移除；既有 dead code 未經要求不動。
- **尺規**：每一行改動都要能直接追溯到使用者的要求。

## 目標驅動、驗證收斂

把任務轉成可驗證的成功條件，據此自主收斂：

- 「加驗證」→「先為非法輸入寫測試，再讓它通過」。
- 「修 bug」→「先寫重現測試，再讓它通過」。
- 「重構 X」→「確保重構前後測試都綠」。
- 多步任務先列簡短計畫，每步標明如何驗證。
- 下任何結論前先對照原始碼／實測／transcript；無法驗證的一律標「假設」，不當事實講。
