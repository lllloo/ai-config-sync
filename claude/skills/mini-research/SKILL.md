# Mini Research — 封頂式多 subagent 比較研究

## 這個 skill 解決什麼

內建 `/deep-research` 品質高,但一次會扇出上百個 subagent——主因是它對**每一條主張**都跑「多票對抗式查證」(N 條主張 × 3 票),乘數效應下 agent 數暴衝、成本難以負擔。

很多實務需求其實是**「比較一批項目、整理成表」**:選型、競品分析、工具/框架/論文 survey。這類任務的正確結構不是「對每句話反覆查證」,而是**「每個項目各自查清楚,再擺在一起比」**。本 skill 就鎖定這個形狀:

- **一個比較項目 = 一個 subagent**,平行深查 → 天然把 agent 數綁在「項目數」上
- **門檻式確認**:項目 **≤ 15** 直接開跑、不打擾使用者;**> 15** 才停下來回報並問使用者要不要縮/分批。多數任務落在 15 以內,所以體感是「直接就跑」,只有真的很大才需要你介入
- **砍掉多票對抗式查證**這個乘數(改由單一 subagent 自行標註不確定處),這是成本能封頂的關鍵
- 最後**由主對話(不另派 agent)彙整成比較表**

換句話說:保留「多 subagent 平行、每項深讀」的品質,拿掉「讓成本乘上去」的部分。

## 何時用、何時不用

- ✅ 用:比較/調查**數個並列項目**,每項要看數個固定維度,最後要一張表或結構化整理。**≤ 15 個直接跑,超過 15 才問使用者。**
- ❌ 不用(改 inline WebSearch/WebFetch):**單一**主題的線性深讀,沒有「並列項目」可拆。派 subagent 只是徒增開銷。
- ❌ 不用(改內建 `/deep-research`):需要對關鍵事實做**對抗式交叉查證**的高嚴謹報告。本 skill 為省成本刻意不做這件事。

## 流程

### 階段 0 — 界定範圍(inline,零 subagent)

先在主對話把三件事定清楚,**不派任何 subagent**,因為這步很便宜且需要跟使用者對齊:

1. **比較項目清單**:使用者已給就沿用;沒給就用 1–2 次 `WebSearch` 生一份候選清單請使用者挑/改。
2. **比較維度**:跟使用者確認每個項目要收哪些欄位(例:授權、價格、star 數、安裝方式、成熟度)。維度先定好,subagent 才能回傳可對齊的結構化結果。

**進階段 1 前的門檻判斷(成本控制點)**:數清單有幾個項目:

- **≤ 15 個 → 直接進階段 1**,不必問。頂多用一句話說「對 N 個項目各派 1 個 subagent,開始」帶過即可,不等回覆。這是為了不用每次都打斷使用者——多數比較任務都落在這個範圍。
- **> 15 個 → 停下來問使用者**,把數字講清楚,例如「清單有 23 個項目,會派 23 個 subagent,要縮到前幾個、分批跑、還是照跑?」,**等使用者回覆**再動。理由:超過 15 個的規模成本明顯升高,值得讓使用者確認一次,而不是默默一次全開。

使用者若指定了上限(例「最多 8 個」),嚴格照辦,只跑最相關的前 N 個,其餘列出待後續。

### 階段 1 — 平行深查(一項目一 subagent)

對清單每個項目派**一個** `general-purpose` subagent(用 Agent 工具,可在同一則訊息內平行送出)。每個 subagent 的任務固定為:

- 針對「這一個項目」跑 **2–4 次 `WebSearch`**、開 **3–6 個關鍵網頁**(`WebFetch`)深讀
- 依階段 0 定好的**維度**回傳結構化結果(每個維度一個值)
- **不確定的值標 `[不確定]`**,並附上判斷依據的來源 URL
- 只回傳精煉後的結構化結果與來源,**不要回傳整頁原文**(控制 context 與成本)

**硬約束**:

- subagent 總數 = 項目數(≤ 15 直接跑);> 15 時 = 階段 0 使用者確認後的數量。使用者指定了上限(例 6 個)就嚴格照辦,只跑前 N 個。
- **不對每條主張再派查證 subagent**。品質靠「單一 subagent 自行標註不確定 + 附來源」承擔,不靠多票乘數。這是本 skill 與 `/deep-research` 的根本差別,不可為了「更保險」私自加回多票查證——那正是成本失控的來源。

給 subagent 的提示模板(依項目替換 `{...}`):

```
研究單一項目並回傳結構化結果,供比較表使用。

## 項目
{item_name}（背景:{item_context}）

## 要收集的維度
{dimensions}  # 例:授權、價格、GitHub star、安裝方式、成熟度

## 方法
- 對這個項目跑 2–4 次 WebSearch,挑 3–6 個最相關的頁面用 WebFetch 深讀
- 每個維度給一個簡潔的值;查不到或沒把握就標 [不確定]

## 回傳格式（僅回傳這些,不要附整頁原文）
- 每個維度一行:「維度名:值」
- 逐值附來源 URL
- 結尾列出標為 [不確定] 的維度名清單
```

### 階段 2 — 彙整成比較表(inline,零 subagent)

所有 subagent 回來後,**由主對話直接彙整**,不另派 synthesis agent:

- 產出一張 **Markdown 比較表**:每列一個項目、每欄一個維度。
- `[不確定]` 的格子原樣標出,不要為了表格好看而抹掉不確定性。
- 表格下方附:
  - **來源清單**(逐項目列出參考 URL,方便查核)
  - **重點觀察**:幾句話點出關鍵差異、取捨建議(這是使用者做決策真正要的)
- 若有項目整批查證失敗(subagent 出錯),明講哪些沒查到,不要假裝表格是完整的。

## 輸出範本

```markdown
## 比較:{主題}

| 項目 | 維度A | 維度B | 維度C |
|------|-------|-------|-------|
| 項目1 | 值 | 值 | [不確定] |
| 項目2 | 值 | 值 | 值 |

### 來源
- 項目1:<url1>、<url2>
- 項目2:<url3>、<url4>

### 重點觀察
- （2–4 句:關鍵差異、對哪種情境建議哪個）
```

## 成本自律(為什麼這些限制重要)

這個 skill 的價值就在「可預期的花費」。以下每一條都直接關係到 agent 數不失控,請理解背後原因、不要繞過:

- **一項目一 subagent + 門檻式確認(15)**:agent 數與項目數線性綁定。≤ 15 直接跑,免打擾;> 15 才問一次。門檻設在 15 是取「多數任務不被打斷」與「大規模成本要有人把關」的平衡點——不是硬砍上限,超過只是多問一句,使用者仍可照跑。
- **不加多票查證**:那是 `/deep-research` 成本的主要放大器。想要更高嚴謹度時,正確做法是**建議使用者改用 `/deep-research`**,而不是在本 skill 裡偷偷把它加回來。
- **subagent 只回結構化摘要、不回整頁**:原文留在 subagent 的 context 裡用完即棄,主對話只收精華,避免 context 爆掉又省 token。
- **階段 0/2 走 inline**:界定範圍與最終彙整不需要獨立 agent,主對話做即可,能省一個算一個。
