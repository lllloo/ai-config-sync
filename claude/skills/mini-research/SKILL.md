# Mini Research — 成本可預期的多 subagent 比較研究

## 這個 skill 解決什麼

內建 `/deep-research` 品質高,但一次會扇出上百個 subagent——主因是它對**每一條主張**都跑「多票對抗式查證」(N 條主張 × 3 票),乘數效應下 agent 數暴衝、成本難以負擔。

很多實務需求其實是**「比較一批項目、整理成表」**:選型、競品分析、工具/框架/論文 survey。這類任務的正確結構不是「對每句話反覆查證」,而是**「每個項目各自查清楚,再擺在一起比」**。本 skill 就鎖定這個形狀:

- **一個比較項目 = 一個 subagent**,平行深查 → 天然把 agent 數綁在「項目數」上
- **門檻式確認**:項目 **≤ 15** 直接開跑、不打擾使用者;**> 15** 才停下來回報並問使用者要**繼續照跑還是縮小範圍**。多數任務落在 15 以內,所以體感是「直接就跑」,只有真的很大才需要你介入
- **砍掉多票對抗式查證**這個乘數(改由單一 subagent 自行標註不確定處),這是成本能壓下來的關鍵
- 最後**由主對話(不另派 agent)彙整成比較表**

換句話說:保留「多 subagent 平行、每項深讀」的品質,拿掉「讓成本乘上去」的部分。

## 何時用、何時不用

- ✅ 用:比較/調查**數個並列項目**,每項要看數個固定維度,最後要一張表或結構化整理。**≤ 15 個直接跑,超過 15 才問使用者。**
- ❌ 不用(改 inline WebSearch/WebFetch):**單一**主題的線性深讀,沒有「並列項目」可拆。派 subagent 只是徒增開銷。
- ❌ 不用(改內建 `/deep-research`):需要對關鍵事實做**多票對抗式交叉查證**的最高嚴謹報告。本 skill 為省成本**預設**不做;若只是想在出表後加**一道輕量把關**,用階段 2.5 的 opt-in 複查(單票、N+1)即可,不必動用 `/deep-research`。

## 流程

### 階段 0 — 界定範圍(inline,零 subagent)

先在主對話把三件事定清楚,**不派任何 subagent**,因為這步很便宜且需要跟使用者對齊:

1. **研究主題/背景**:一句話界定「在比什麼、為了什麼決策」。這會成為輸出表的 `{主題}`,也是各 subagent 提示裡 `{item_context}` 的來源——先定好,subagent 才知道用什麼角度深查、彙整時才有共同脈絡。
2. **比較項目清單**:使用者已給就沿用;沒給就用 1–2 次 `WebSearch` 生一份候選清單請使用者挑/改。
3. **比較維度(連同「值的格式」一起定)**:跟使用者確認每個項目要收哪些欄位,並為每欄釘死**值要用什麼單位/格式**——這是比較表能真正對齊的關鍵。例:「價格→月費 USD」「star→約略數字(如 12k)」「授權→SPDX 代號(MIT/Apache-2.0)」「安裝方式→一句話」。格式先定好,各 subagent 才不會一個回『免費』、一個回『$0/mo』、一個回『有 free tier』,湊進表裡變成蘋果比橘子。

**進階段 1 前的門檻判斷(成本控制點)**:數清單有幾個項目:

- **≤ 15 個 → 直接進階段 1**,不必問。頂多用一句話說「對 N 個項目各派 1 個 subagent,開始」帶過即可,不等回覆。這是為了不用每次都打斷使用者——多數比較任務都落在這個範圍。
- **> 15 個 → 停下來問使用者**,把數字講清楚,例如「清單有 23 個項目,會派 23 個 subagent,要**照跑全部還是縮到最相關的前幾個**?」,**等使用者回覆**再動。理由:超過 15 個的規模成本明顯升高,值得讓使用者確認一次,而不是默默一次全開。(分批跑不省成本——同樣項目數就是同樣 subagent 總數,故不列為選項;真正的取捨只有「照跑」或「縮小範圍」。)

使用者若指定了上限(例「最多 8 個」),嚴格照辦,只跑最相關的前 N 個,其餘列出待後續。

### 階段 1 — 平行深查(一項目一 subagent)

對清單每個項目派**一個** `general-purpose` subagent(用 Agent 工具,可在同一則訊息內平行送出)。每個 subagent 的任務固定為:

- 針對「這一個項目」跑 **2–4 次 `WebSearch`**、開 **3–6 個關鍵網頁**(`WebFetch`)深讀
- 依階段 0 定好的**維度**回傳結構化結果(每個維度一個值)
- **不確定的值標 `[不確定]`**;本項目用到的來源 URL 結尾統一列一份即可(不逐值標)
- 只回傳精煉後的結構化結果與來源,**不要回傳整頁原文**(控制 context 與成本)

**硬約束**:

- subagent 總數 = 項目數(≤ 15 直接跑);> 15 時 = 階段 0 使用者確認後的數量。使用者指定了上限(例 6 個)就嚴格照辦,只跑前 N 個。
- **不對每條主張再派查證 subagent**。品質靠「單一 subagent 自行標註不確定 + 附來源」承擔,不靠多票乘數。這是本 skill 與 `/deep-research` 的根本差別,不可為了「更保險」私自加回多票查證——那正是成本失控的來源。

給 subagent 的提示模板(依項目替換 `{...}`):

```
研究單一項目並回傳結構化結果,供比較表使用。

## 項目
{item_name}（背景:{item_context}）

## 要收集的維度（含每欄要求的值格式,務必照格式回）
{dimensions}  # 例:授權→SPDX 代號、價格→月費 USD、star→約略數字、安裝方式→一句話

## 方法
- 對這個項目跑 2–4 次 WebSearch,挑 3–6 個最相關的頁面用 WebFetch 深讀
- 每個維度給**一個照指定格式、能跨項目直接比**的值;查不到或沒把握就標 [不確定]
- 若這個項目本身有意外狀況(已停更/改名/被併購/根本不符背景),一句話點出——別硬填欄位假裝正常

## 回傳格式（僅回傳這些,不要附整頁原文）
- 每個維度一行:「維度名:值」
- 結尾統一列出本項目用到的來源 URL（彙整成一份即可,不必逐值標）
- 最後列出:標為 [不確定] 的維度名清單,以及(若有)上面提到的意外狀況
```

### 階段 2 — 彙整成比較表(inline,零 subagent)

所有 subagent 回來後,**由主對話直接彙整**,不另派 synthesis agent:

輸出採**倒金字塔**:決策先行、數據殿後。順序固定為「分級三段 → 完整對照表 → 來源 → 重點觀察」。

- **① 分級三段(導讀,放最前)**:把項目分 **首選 / 可選 / 不推薦** 三組,每組列項目 + 一句理由,回扣階段 0 的決策脈絡。理由是**綜合各維度的判斷**,不是照抄對照表某格的值(如「價格 $0」)——那只是重述數據、缺乏決策增值。分級**不確定就講明、別硬分**;某段空著(如沒有該淘汰的)就**省略該段**,不硬湊。這一段就是使用者做決策真正要的,先給。
- **② 完整維度對照表(數據層)**:別為了好看把它砍成一句話——**逐維度對照是本 skill 的核心價值**,一定保留。
  - **落表前先對齊格式**:值沒照約定單位/格式的(如混用『免費』與『$0/mo』)在此統一成一種寫法;拿不準時保留原值並標註,不臆造。
  - `[不確定]` 原樣標出,不為了表格好看抹掉不確定性。
  - **瘦身讓表好讀**:欄數盡量收在 **5 欄以內**、格子放**短標/數字/符號**(長說明移到表下註腳);維度真的多(**>6 欄**)或值偏長時,**改用卡片式**(每項一個小標題 + 條列)取代寬表,別硬塞會橫向溢出的大表。
- **③ 來源清單**:逐項目列 URL,可順帶標**來源品質**(primary/secondary/blog),方便查核與判斷可信度。
- **④ 重點觀察(caveats + 待補)**:建議已在分級三段給過,這裡**不複述**——只聚焦**還不確定/來源弱/有時效性**之處,以及**浮現但沒答的問題**(1–3 點),供使用者判斷可信度與後續方向。
- 若有項目查不到或 subagent 出錯,明講哪些沒查到、哪些有意外狀況(停更/改名等),不假裝完整。

### 階段 2.5 — 選配:彙整後複查(opt-in,預設不做)

**預設不做。** 本 skill 預設零查證以壓成本;只有當這批比較的**事實錯誤代價高**(選型涉及合規/安全/簽約付錢)時,才在彙整成表後補**一道**複查。做法照 Anthropic multi-agent research 的實測結論設計:

- **形狀**:派**一個**複查 subagent(**不是**多票 panel——Anthropic 實測「單一 judge + 單一 prompt 輸出分數」比 multi-judge panel 更穩、更貼合人類),用**一個 prompt** 對**整張表**打分,不逐項、不逐主張。成本 = **N+1**,不是翻倍的 2N。
- **評分四維**(借 Anthropic 的 rubric),各輸出 0.0–1.0 + 整體 pass/fail:
  - **factual accuracy**:表中的值和它掛的來源對得上嗎
  - **citation accuracy**:引用歸屬對嗎(有沒有掰來源、張冠李戴)
  - **completeness**:每個該有值的格子都填了嗎(還是有該查沒查的)
  - **source quality**:偏重一手來源嗎
- **兩段強度**:
  - **預設(便宜)**:複查 agent 只吃「表 + 各項來源摘要」,核對表值與來源是否自洽、跨項口徑是否一致。
  - **高風險(較貴)**:要求複查 agent **重開關鍵來源比對原文**——只吃摘要會踩 Anthropic 點名的「傳話遊戲(game of telephone)」失真,連「來源本身被讀錯」都要抓時才升級。
- 複查回來後,主對話據以修正表格,存疑格在表中就地標註;並把 **pass/fail 與四維分數放在標題下、分級三段之前**,讓使用者一眼知道這張表過沒過複查。

**這一步不違反成本自律**:被禁的是「對每條主張多票對抗查證」(O(N·C·V) 乘數);這裡是彙整後**一次、單票、看整張表**的歸屬核對(N+1),成本仍與 N 線性。要最高嚴謹度的多票交叉查證,仍改用 `/deep-research`。

複查 subagent 的提示模板(替換 `{...}`):

```
複查一張比較研究表的品質,回傳評分與需修正處。

## 待複查的表與來源
{comparison_table}
{per_item_sources}

## 複查方式
- 對整張表打分,不必逐項派查證。四個維度各給 0.0–1.0 + 一句依據:
  - factual accuracy:表中的值與其來源是否相符
  - citation accuracy:每個值是否掛了對得上的來源,有無掰來源/張冠李戴
  - completeness:是否有欄位該有值卻空著或含糊
  - source quality:是否偏重一手來源
- 〔僅高風險模式加這行〕重開這些關鍵來源比對原文,不要只憑上面的摘要:{key_urls}
- 整體 pass/fail:任一維 < 0.6 記 fail

## 回傳格式
- 四維各一行:「維度:分數 — 依據」
- 整體:pass / fail
- 需修正的具體格子清單:「項目×維度:問題 + 建議值(或改標存疑)」
```

## 輸出範本

```markdown
## 比較:{主題}

### 首選 — {項目}
{一句理由,回扣決策脈絡}

### 可選 — {項目}
{一句理由:適合什麼情境}

### 不推薦 — {項目}（{原因,如已停更}）
（沒有該淘汰的項目時,整段省略）

### 完整對照
| 項目 | 維度A | 維度B | 維度C |
|------|-------|-------|-------|
| 項目1 | 值 | 值 | [不確定] |
| 項目2 | 值 | 值 | 值 |

### 來源
- 項目1:<url1>、<url2>(primary)
- 項目2:<url3>(blog)

### 重點觀察
- caveats:{不確定/來源弱/有時效性之處}
- 待補:{浮現但沒答、需再查的問題}
```

維度多(>6 欄)或值偏長時,把「完整對照」表換成**卡片式**:

```markdown
### 完整對照
#### {項目1}
- 維度A:值 · 維度B:值
- 維度C:[不確定] · 維度D:值

#### {項目2}
- 維度A:值 · 維度B:值
```

## 成本自律(不要繞過的三條線)

這個 skill 的價值就在「可預期的花費」。上面的限制不是保守,是成本模型本身,理解後別繞過:

- **agent 數 = 項目數,別再乘上去**:一項目一 subagent 讓成本與項目數線性綁定;「對每條主張多票查證」正是 `/deep-research` 成本暴衝的乘數,本 skill 刻意不做。想要那種嚴謹度時,**建議使用者改用 `/deep-research`**,不要在這裡偷偷加回來。(唯一例外是階段 2.5 的 opt-in 複查:那是彙整後**一次、單票、看整張表**的 N+1 核對,不是「對每條主張乘上票數」,兩者本質不同——它是 opt-in、預設關,不是偷偷加回乘數。)
- **15 是「多問一句」的門檻,不是硬上限**:設在「多數任務不被打斷、大規模才有人把關」的平衡點;超過只是停下確認,使用者仍可照跑。
- **原文留在 subagent、只回摘要**:整頁用完即棄,主對話只收精華——省 token 又不撐爆 context。階段 0/2 走 inline 同理,能省一個 agent 是一個。
