# mem0 長期記憶 — 自動記錄與回憶跨專案知識

## 定位與分工

mem0 是 AI 的**跨工具、跨專案長期記憶**（經 MCP 工具 `add_memory`／`search_memories`／`update_memory` 等存取）。分工原則：

- 環境中有 mem0 工具時，**跨專案通用知識一律進 mem0**，不寫入各 harness 自帶的檔案式 memory（如 Claude Code 的 `~/.claude/projects/*/memory/`）——那層只保留專案脈絡類記憶（進行中工作、專案目標等本就不該進 mem0 的內容）。
- 環境中沒有 mem0 工具（未裝該 MCP 的裝置或工具）：**靜默跳過**，退回該環境原生的記憶機制，不報錯、不提醒安裝。
- 與使用者自己的筆記系統（如 Obsidian vault）不衝突：筆記是**使用者**的知識庫，mem0 是 **AI** 的工作記憶；查使用者的既有研究走該環境提供的筆記查詢管道（如 ask-vault skill，若有），不查 mem0。

## 自動記錄：四類觸發

發現當下即存，**不等對話結束**。一次一條：

| 類型 | 內容 | metadata type |
|------|------|---------------|
| 技術知識與坑 | 跨專案通用、**已驗證**的技術事實、API 行為、版本地雷、除錯心得 | `tech` |
| 使用者偏好與回饋 | 使用者對工作方式的糾正與偏好，**含為什麼與怎麼套用** | `preference` |
| 工具與流程心得 | CLI 技巧、被證實有效的工作流程模式 | `workflow` |
| 決策原則 | 跨專案通用的架構／選型判斷原則（何時選 A 不選 B 的理由） | `decision` |

## 排除清單

一條錯誤或含機密的記憶會被之後**所有** session、**所有**工具讀到，污染永久且跨專案，所以**機密與未驗證的事實寧可漏記**；其餘邊界案例（該不該算通用知識）往「存」判即可，存錯一條通用知識的成本遠低於漏記。

- **專案細節**：程式碼結構、檔案路徑、特定 repo 的事實、可從 repo 本身（CLAUDE.md／git 歷史）推導的內容。
- **機密**：token、API key、密碼、環境變數值——**任何情況都不存**，改寫過也不存。
- **一次性脈絡**：只對本次對話有意義的內容。
- **他人個資**：他人的姓名、email 等聯絡資訊。

## 直接存，不做存前改寫

**存入不需要先改寫**：知識當下怎麼講的就怎麼存，不必為了去識別化或平台中立而重述一遍——那層儀式讓「順手記一筆」變成一道工序，實際結果是常常乾脆不記。

**只記已經發生的，不生成新內容**：記憶內容限於對話裡實際出現過的事實與使用者實際說過的話，用最貼近原意的說法寫下即可。不補充推論、不延伸解釋、不歸納出使用者沒表達過的原則、不替他預設「為什麼」與「怎麼套用」——那些是你的臆測，一旦入庫就會被之後所有 session 當成使用者的真實意圖。原話夠清楚就直接用原話。

- 提到專案名、檔案路徑、工具名都照寫，不特意抽掉。
- 唯一硬界線仍是上面的排除清單（機密任何情況不存）。
- 行有餘力時把句子寫得可遷移（「Node 18 的 X API 有 Y 行為」勝過綁死單一 repo 的敘述），但**不因寫不出通用句而不存**。

## add_memory 呼叫慣例

- `text` 為**單句完整事實**，一次一條，不把多條打包成一次呼叫。
- **一律帶 `infer: false`**：預設 `true` 會讓 server 端 LLM 再抽取／改寫一次，入庫的句子未必是你寫的那句；`false` 才是原文照存，符合「只記已經發生的，不生成新內容」。
- **不帶 `user_id`／`agent_id`／`run_id`，交由 server 預設 user scope**——工具說明寫「requires at least one of user_id/agent_id/run_id」，但那是 server 未設預設 scope 時的要求；帶上 `agent_id` 等 ID 會把記憶切進單一工具／單次執行的隔間，其他工具回憶時搜不到，跨工具共用的目的直接失效。`search_memories` 同理不加 agent 類 filter（server 會自動注入 user scope）。
- 繁體中文撰寫，技術術語保留英文。
- `metadata` 帶 `{ "type": "tech" | "preference" | "workflow" | "decision" }` 供日後過濾。
- 相對時間轉絕對日期（「上週」→ 實際日期）。
- 寫入為非同步（回 event_id）：**不** poll `get_event_status`，記錄後即繼續原任務，不打斷工作流。
- 疑似已存過的知識先 `search_memories` 查重：已存在且內容有更新用 `update_memory`，完全相同則不動作，不重複新增。
- 回憶時發現既有記憶已被後續事實推翻：`update_memory` 修正；無法修正（整條就是錯的）用 `delete_memory` 刪**該條**。**任何情況不呼叫 `delete_all_memories`**——批次刪除不可逆，且會波及其他工具、其他對話存下的所有記憶。

## 自動回憶

- 開始處理「過往可能累積過知識」的主題（除錯特定技術、選型比較、涉及使用者習慣的決策）時，先 `search_memories`（自然語言 query）再作答。
- 使用者說「我之前」「上次」等回溯語意時**必查**。
- 查詢失敗或無結果就正常繼續作業，不重試多次、不向使用者報錯。
